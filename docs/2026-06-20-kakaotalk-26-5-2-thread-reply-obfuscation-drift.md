# KakaoTalk 26.5.2 thread reply send failure

## Summary

KakaoTalk 26.5.2 updated the obfuscated `ChatSendingLog` method layout. The Iris bridge reply path was trying to write thread reply metadata by calling guessed setter names through reflection. A newly added `H1(int)` method had the same one-`int` signature as the expected `scope` setter, so reflection succeeded while writing the value to the wrong semantic target.

The intended target was not `H1(int)` itself. The intended target was:

```text
ChatSendingLog.Z   -> scope
ChatSendingLog.V0  -> threadId
```

The accidental target in KakaoTalk 26.5.2 was:

```text
ChatSendingLog.H1(int) -> multiUploadSequence
```

This caused the bridge to record a send as completed even though the `scope` field stayed `0`. KakaoTalk then did not treat the log as a valid thread reply, so reply/quoted messages were missing from the room while ordinary text sends still worked.

## Affected Path

The failing bridge code path was:

```text
bridge/src/main/java/party/qwer/iris/imagebridge/runtime/reply/ReplyMarkdownSendingLogAccess.kt
  writeThreadMetadata(...)
```

This function is used by reply markdown and threaded send paths to graft thread metadata onto a KakaoTalk `ChatSendingLog` object.

The metadata has two required pieces:

```text
threadId     The original message/thread identifier that the reply belongs to.
threadScope  The KakaoTalk visibility scope for chat room/thread-detail display.
```

Without both values being written to the right KakaoTalk fields, a send can appear locally successful in the bridge while KakaoTalk does not render or ship it as the intended reply.

## What The Bridge Was Trying To Target

The bridge was trying to reproduce the important part of KakaoTalk's normal message composition flow:

```kotlin
sendingLog.scope = threadScope
sendingLog.threadId = threadId
```

In KakaoTalk's decompiled `ChatSendingLog`, those stable fields are:

```text
scope    -> Z
threadId -> V0
```

The target was therefore the semantic pair `scope` and `threadId`, not any particular obfuscated method name.

The confusion came from the old implementation using guessed setter names:

```text
call guessed int setter   -> expected to set scope
call guessed Long setter  -> expected to set threadId
```

This was fragile because the method names are generated by obfuscation and can move between KakaoTalk releases.

## KakaoTalk Scope Semantics

KakaoTalk models chat log visibility using `ChatLogScope`. In KakaoTalk 26.5.2 the relevant values are:

```text
1 = ONLY_CHAT_ROOM
2 = ONLY_THREAD_DETAIL
3 = CHAT_ROOM_AND_THREAD_DETAIL
```

The helper logic is bit-based:

```text
isVisibleInChatRoom     -> (scope & 1) != 0
isVisibleInThreadDetail -> (scope & 2) != 0
```

For a reply/threaded message that must be visible in both contexts, `scope = 3` is the meaningful value. If `scope` remains `0`, neither visibility bit is set. That is why the failure mode was not an authentication error and not a mux transport error: KakaoTalk received a `ChatSendingLog` whose thread visibility metadata was invalid.

## Version Mapping

The relevant setter names changed across KakaoTalk versions:

```text
KakaoTalk 26.4.2
  J1(int)   -> scope setter
  L1(Long)  -> threadId setter

KakaoTalk 26.5.2
  L1(int)   -> scope setter
  N1(Long)  -> threadId setter
  H1(int)   -> multiUploadSequence setter
```

The field names stayed stable in the inspected versions:

```text
Z   -> scope
V0  -> threadId
```

This is why the fix writes fields directly instead of tracking obfuscated method names.

## Failure Mechanism

The old bridge behavior can be reduced to this shape:

```kotlin
runCatching {
    sendingLog.getMethod("H1", Int::class.javaPrimitiveType)
        .invoke(sendingLog, threadScope)
}.getOrElse {
    sendingLog.getDeclaredField("Z").setInt(sendingLog, threadScope)
}
```

The expected behavior was:

```text
H1(int) absent or correct scope setter
  -> scope gets threadScope

or

H1(int) missing
  -> fallback writes Z directly
```

KakaoTalk 26.5.2 introduced the bad middle case:

```text
H1(int) exists
H1(int) accepts the same int argument
H1(int) does not set scope
H1(int) sets multiUploadSequence
```

Reflection therefore returned success. Because there was no exception, the fallback write to `Z` never ran.

The resulting object state was effectively:

```text
Expected:
  Z(scope)              = threadScope
  V0(threadId)          = threadId
  multiUploadSequence   = unchanged

Actual in the failure:
  Z(scope)              = 0
  V0(threadId)          = threadId, depending on the second setter/fallback path
  multiUploadSequence   = threadScope
```

This is a silent semantic failure: the reflection call is mechanically valid but semantically wrong.

## Why Plain Text Still Worked

Plain text sends do not need thread reply metadata. They can be sent with the normal text path without setting `scope` and `threadId` for a reply.

The failure was specific to reply/quoted/threaded sends because those paths require:

```text
scope    non-zero and compatible with thread visibility
threadId positive and pointing at the target thread/message
```

When `scope` stayed `0`, the reply metadata was incomplete even if the bridge logged the send attempt as completed.

## Why The Logs Were Misleading

The bridge log reflected the state of the bridge operation, not the semantic correctness of KakaoTalk's internal `ChatSendingLog` fields.

From the bridge's perspective:

```text
Reflection method lookup succeeded.
Reflection invoke returned without exception.
The send call path continued.
```

So the bridge could log a completed text send.

From KakaoTalk's perspective:

```text
scope = 0
thread reply visibility is not valid
the log is not handled as the intended thread reply
```

This explains the observed symptom:

```text
Bridge logs: send completed
KakaoTalk room: reply message missing
Authentication/HMAC: not involved
Mux bridge transport: connected and executing
Actual failure layer: KakaoTalk internal send-log grafting
```

## Fix

The fix removed setter-name guessing for thread metadata.

Current intended write path:

```kotlin
setThreadField(sendingLog, "Z", threadScope)
setThreadField(sendingLog, "V0", java.lang.Long.valueOf(threadId))
```

The helper walks the class hierarchy because `getDeclaredField` only checks the current class and does not search superclasses. If the field is absent, it raises a diagnostic `NoSuchFieldException` that includes the field name and class hierarchy context.

This is intentionally fail-loud. If a future KakaoTalk version renames or removes `Z`/`V0`, the bridge should fail with an actionable error instead of silently writing the wrong field.

## Regression Coverage

The regression test models the 26.5.2 trap:

```kotlin
var V0: Long? = null
var Z: Int = 0
var multiUploadSequence: Int = -1

fun H1(value: Int) {
    multiUploadSequence = value
}
```

The expected assertions are:

```text
V0 == threadId
Z == threadScope
multiUploadSequence == -1
```

The important assertion is that `multiUploadSequence` remains unchanged. That proves `writeThreadMetadata` is not calling the version-shifted `H1(int)` trap.

Additional coverage verifies:

```text
direct field writes work when setters are absent
inherited Z/V0 fields are found through superclass traversal
missing thread fields raise a diagnostic NoSuchFieldException
```

## Maintenance Rule

For thread reply metadata, prefer stable semantic fields over obfuscated method names.

Do not add a fallback that scans one-argument `int` methods or restores guessed method names for `scope`. A method call that succeeds is not enough evidence that the correct field was written.

When KakaoTalk updates:

```text
1. Confirm ChatSendingLog.Z still maps to scope.
2. Confirm ChatSendingLog.V0 still maps to threadId.
3. Confirm the reply regression test still protects against method-signature traps.
4. If either field moves, update the field map and keep the failure mode diagnostic.
```

The main invariant is:

```text
writeThreadMetadata must write the semantic thread fields and must not call unrelated setters that only match by signature.
```
